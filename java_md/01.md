## 📌 1. 개발 환경 & Git/GitHub 실전 트러블슈팅

### 🚨 [대원칙] Git Bash 실행 시 최초 저장소 이동 필수
* **발생 에러**: `fatal: not a git repository (or any of the parent directories): .git`
* **원인**: Git Bash를 처음 켜면 컴퓨터의 홈 최상위 경로에서 시작함. `.git` 숨김 폴더가 없는 '엉뚱한 위치'에서 깃 명령어를 치면 Git이 저장소를 찾지 못해 발생함.
* **해결법**: 반드시 `cd` (Change Directory) 명령어를 사용해 **진짜 코드가 들어있는 깃 저장소 폴더까지 한 단계씩 진입**한 후에 명령어를 입력해야 함.
* **실전 경로 이동 치트키**:
  ```bash
  # 1. 바탕화면(Desktop) 폴더로 이동
  cd desktop

  # 2. 강의/과제 소스코드가 있는 작업 폴더(lect)로 이동
  cd lect

  # 3. (팁) 한 번에 이동하고 싶을 때
  cd desktop/lect
  ```

### 📦 Git 핵심 전송 흐름 (택배 비유)
* **`git add .`**: 현재 작업 공간(Working Directory)에서 수정된 파일들을 커밋 대기 상태로 전환하여 **스테이징 영역(Staging Area/Index)**에 등록하는 명령어
* **`git commit -m "msg"`**: 스테이징 영역에 등록된 파일들의 상태를 스냅샷(Snapshot)으로 캡처하여 **로컬 저장소(Local Repository)**에 영구 기록하는 명령어
* **`git push`**: 변경 사항들이 하나의 고유한 해시(Hash) 값을 가진 '커밋 객체'로 생성되어 로컬 버전 히스토리에 포함됨

### 🛠️ 실전 치트키 명령어
* **유저 이름 변경 후 연결 오류 (`not found`) 해결**
  ```bash
  git remote set-url origin https://github.com
  ```
* **윈도우와 맥/리눅스의 줄바꿈 경고 (`LF will be replaced by CRLF`) 해결**
  ```bash
  git config --global core.autocrlf true
  ```
* **구형 계정 정보 충돌 자격 증명 청소**
  * 윈도우 검색창 ➡️ `자격 증명 관리자` ➡️ `Windows 자격 증명` ➡️ `git:https://github.com` 찾아서 **제거**

## ☕ 2. Java 핵심 철학 & 메모리 메커니즘

### 🔒 `final` 자물쇠의 3단계 확장성
1. **변수 앞**: 값을 변경할 수 없는 **상수**로 고정
2. **메서드 앞**: 자식 클래스에서 이 함수를 내 입맛대로 리모델링(**오버라이딩**)하는 것을 절대 금지
3. **클래스 앞**: 이 클래스를 복사·조립하여 자식 클래스를 만드는 것(**상속**)을 통째로 불허

### 🔄 오버라이딩(Overriding)의 본질
* **정의**: 부모가 물려준 함수(메서드) 위에 올라타서 내부 알맹이 내용을 완전히 덮어써 버리는 기술
* **존재 이유**: 자식이 좋아서가 아니라 **이들을 통제하는 메인 프로그램이 편해지려고 사용**. 메인 프로그램은 자식들이 내부를 어떻게 고쳤는지 몰라도, 부모가 정한 공용 리모컨 버튼(`.attack()`, `.volumeUp()`) 하나만 누르면 자식들이 알아서 각자의 고성능 기능을 발휘함 (다형성)

### ⚡ `static` 정적 할당 vs `new` 동적 할당
* **`static` (정적 할당)**
  * **본질**: 프로그램이 **켜지는 순간** 메모리(Static Area) 공간을 즉시 자동 배정
  * **철칙**: `main` 함수는 켜지자마자 할당되므로, 아직 메모리가 존재하지 않는 일반 메서드를 호출할 수 없음. 따라서 호출할 메서드 앞에도 `static`을 붙여 메모리를 먼저 만들어 두어야 함
* **`new` (동적 할당)**
  * **본질**: 프로그램 **실행 도중** 필요한 순간 실시간으로 힙(Heap) 메모리 공간을 빌려 씀 (C언어의 `malloc`과 동일)
  * **장점**: 미래에 어떤 크기의 데이터가 올지 모르는 불확실한 상황에서 메모리를 유연하게 조절하고, 다 쓴 공간은 자바 가비지 컬렉터가 자동으로 수거하여 메모리를 극도로 절약함

## 🧠 3. 프로그래머스 실전 알고리즘 분석

### 🗂️ [완주하지 못한 선수] - 시간 복잡도 최적화
* **문제 한계**: 이중 루프 구사 시 $O(N^2)$으로 데이터 10만 개 기준 **100억 번 연산**이 되어 무조건 시간 초과 탈락
* **정답 로직 ($O(N \log N)$)**: 두 배열을 가나다순으로 **정렬(`sort`)**한 뒤, 단 한 바퀴만 돌며 1:1 대조하다가 이름이 달라지는 그 즉시 `return`하여 범인 검거
* **C++ vs Java 문법**: C++ 벡터 크기는 `.size()`, 자바 일반 배열 크기는 `.length` 사용

### 🌲 [타겟 넘버] - DFS(깊이 우선 탐색) 재귀 스택의 마법
* **핵심 로직**: 숫자를 순방향으로 전진시키며 '더할까(+) 뺄까(-)' 결정하는 고정된 2방향 트리 구조
* **참조 전 리턴(Return)의 비밀**:
  * 함수 중간에 인덱스를 인위적으로 키우지 말고, 다음 재귀 함수를 호출하는 매개변수 자리에 직접 `index + 1`을 던져야 함
  * 그래야 존재하지 않는 배열 방을 참조하기 직전에, 맨 윗줄 종료 조건문(`index == length`)에서 먼저 걸려 안전하게 리턴(탈출)할 수 있음

### 📐 [최소직사각형] - 2차원 배열 완전탐색
* **입력 구조**: `[[60, 50], [30, 70]]` 형태는 명확한 **2차원 배열(`int[][]`)** 구조
* **수학적 본질**: 지갑의 크기 = [가로 1등] × [세로 1등]
* **정답 로직**: 모든 명함의 **더 큰 값을 가로방으로 몰아넣고, 더 작은 값을 세로방으로 가둔다.** 큰 숫자를 이미 더 큰 숫자가 대장으로 버티고 있는 방으로 던져버리면, 그 방의 최댓값은 변하지 않으면서 반대쪽 방의 최댓값만 쏙 낮출 수 있어 무조건 가장 작고 날씬한 지갑이 완성됨 ($O(N)$으로 해결)
